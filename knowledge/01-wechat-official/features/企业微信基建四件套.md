# 企业微信冷启动基建 · 四件套（功能特性层）

> 综合自公开的企业微信官方能力与私域实战通行打法。
> 定位：可公开的「微信生态 / 功能特性层」，给 `ops-as-ad-funnel`、`siyu-qunfa`、`siyu-huashu` 提供企微冷启动的默认框架。
> 真实操盘路径 / 话术 / 阈值 / 报价是护城河，在 `knowledge/03-majia-sop/`（私有），此处只给脱敏方法论。
> 行业应用（餐饮加微钩子、活动文案、迁移剧本）见 `knowledge/02-industry/catering/`。

## 〇、认知前提：企微 vs 个微，先讲透再动手

- 个人微信回复最有「活人感」，但**复制不了**，靠堆人、多开维持。
- 企业微信牺牲一部分人味，换来**可复制 + 自动化 + 官方合规**——连锁扩店的地基。
- **搭台唱戏**：企微基建是地基和舞台，内容与营销系统才是主角。地基决定楼能盖几层。
- 操作前先跟经营者把「企微不是完美的、它有取舍」讲清楚，损失的人味后续用活动补回，避免预期错位。

> 判断动作时对齐公理「私域即公关」：企微把关系场做成可复制的基础设施，但每次触达仍在存钱或取钱。

## 一、四件套（冷启动按序搭，缺一不成地基）

一套完整的企微承接底座就四个模块，**顺序不能乱**（后一件依赖前一件）：

| 序 | 模块 | 官方路径（概念） | 作用 |
|---|---|---|---|
| ① | **好友码** | 客户联系 → 加客户 → 联系我 → 新建二维码 | 一切的源头，所有物料都贴它 |
| ② | **对外资料页** | 我的企业 → 通讯录管理 → 对外展示 | 顾客可见别名 + 自定义信息 + 跳转位 |
| ③ | **好友欢迎语** | 加客户 → 欢迎语 | 加好友后自动触发一次，引导进群 |
| ④ | **群活码 + 进群欢迎语** | 客户联系 → 客户群 → 建群码 | 自动接续建群 + 进群承接 |

### 件① 好友码 —— 先加好友，再进群

- 先加单个成员，备注一个**顾客可见别名**（如「XX 店长」）。
- **为什么好友优先于进群**：加好友后可通过朋友圈 + 私聊 + 群消息**三通道同时触达**，曝光面最大。这是好友码作为源头的核心理由。

### 件② 对外资料页 —— 让顾客有正规感又有落点

- 别名：顾客侧显示「XX 店长」而非本名 / 法人全名（离岗后没人找得到是反面）。
- **自定义字段**：加一条「优惠信息」，挂已有的团购 / 点评链接。
- 三个跳转位：查看优惠 / 加入社群 / 特别备注，对标成熟账号逐格配。
- 配后需「组织架构 → 编辑成员 → 保存」并等约 1 分钟刷新；顾客端有缓存属正常。

### 件③ 好友欢迎语 —— 自动、只触发一次

公式：`顾客昵称变量` + 认领身份 + 一个钩子 + 一个入群动作。

- 插入「客户昵称」变量，开头「XX 你好」。
- 带一点本地方言 + 少量表情，**控制在几百字内**。
- 它是**中间态**，目的是引导进群，不是让顾客读大段文字。
- 系统自动触发一次，无需像个人微信逐个私发。

> 对齐 `siyu-huashu`：欢迎语先认领身份和价值，再提一个轻动作。

### 件④ 群活码 + 进群欢迎语 —— 官方自动接续

- **群活码**是一个**永不改变的入口码**：单群满 200 人（企微群上限）后，官方「客户群」能力**自动接续新建下一个群并分流**，群名 / 群主 / 欢迎语按**群模板**自动继承。这是个人微信不具备的连锁能力，属企微官方白名单功能。
- 群模板可设：起始群号、禁改群名、禁成员互加、指定管理员。
- 「点击进群链接」体验优于纯二维码，但原生只出码；要出链接需第三方 SCRM，结果一致。
- **进群欢迎语二选一**：
  - A：每 120 分钟触发一次，提醒该时段新进成员。
  - B（多数品牌选）：不设欢迎语，改用**群名前缀「进群看公告」**，营销信息统一维护在群公告，靠一线引导。更少打扰，代价是要主动引导顾客查看公告。

## 二、认证：企业名 = 正规军

- **常见卡点**：想用的对外名称是**他人已注册商标**（尤其热门品类词），认证被驳回。
- **解法**：基于**工商字号简称**命名，或提交**自有商标证书 / 授权**。
- **命名铁律**：对外品牌名是长久的事，先定死——名称别过长、热门词多拿不下、用能持有的字号；顾客看朋友圈 / 群消息第一反应就是这个名。
- 认证后店员消息背后显示企业名，更像正规军，但少一分活人感，用活动补。

## 三、老客迁移：不一刀切

- 从个人微信群迁到企微群，**不要一上来宣布解散老群**（伤感情）。
- 动作链：带钩子的进群物料发老群 → 通知全员福利转移 → 顾客扫码加企微好友 → 欢迎语自动推群码 → 进群。全靠好友码一个源头。
- 先用**新群专属钩子**引一波；若老客盘只进来少数，再谈解散，给缓冲。
- 对齐 `reactivation-playbook`：迁移是召回动作，先给理由和价值，再提迁移。

## 四、SCRM 选型阶梯：跑通再上系统

企微原生免费但配推送 / 看数据繁琐。三档决策：

1. **首选**：团购 / 会员平台系统是否自带社群模块——先问商务含不含、是否额外付费，**数据能打通的优先选**。
2. **次选**：第三方 SCRM 标准版（有免费试用），补齐渠道活码 / 群管理 / 群发 / 数据看板 / 一线拉新激励。
3. **兜底**：企微原生，先玩熟。

- **一键迁移**：升级第三方时好友 / 群都在（挂企微账号），多支持一键搬码；就算重建，物料旧码盖新码即可，代价低。
- **建议**：前期先把原生玩熟、跑通，再按报价决定是否上系统，不必一步到位。

## 五、两个成本 / 数据口径

- **加人成本**：企微 2000 客户以内免费，超出后每客约一毛/年（量级：2 万≈2 千、5 万≈5 千）。进群不加好友也占一个额度。对几千顾客体量成本极低，复购收益远覆盖。
- **标签口径**：顾客标签必须**绑定真实消费行为**才有意义。企微自身打标签缺抓手（对消费无感知）；有效标签依赖会员 / 团购系统读取消费数据，SCRM 再读该标签。别在没有消费数据时空打标签。

## 六、客服提效（合规范围内）

- **快捷回复**：常用应答（发票 / 营业时间）沉淀成分组模板，一线会话侧一键发送。
- **关键词自动应答**：高频重复问题可用群内关键词自动承接，减少人工。具体实现路径与边界以私有 SOP 为准，公开层只保留「高频问题关键词化」这一方法。

## 七、合规提示（对齐合规红线）

- 群自动接续、客户联系 / 客户群 / 群发均走**企微官方白名单能力**；不碰任何违反企微协议的第三方外部工具类手段（禁用清单以 [`../compliance/redlines.md`](../compliance/redlines.md) 为准）。
- 迁移与活动文案走真实条件，不设欺骗式门槛，过合规扫描（见 `knowledge/01-wechat-official/compliance/redlines.md`）。
- 收集顾客信息需明示授权。
